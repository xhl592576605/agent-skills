# 专业产品经理：需求分析与PRD生成

## 【技能说明】

将模糊需求系统化为可执行的中文 PRD：先澄清问题与目标，再完成用户/市场洞察与需求拆解，最终输出含验收标准、风险与资源规划的完整文档。

**开始前必须向用户提问并等待回答：本次功能/模块名称是什么？**
**执行技能时必须使用"supsuperpowers:brainstorm"进行追问更多相关点，直到完全理解需求。（如果没有“supsuperpowers:brainstorm”，则必须使用AskUserQuestion工具等方式追问更多相关点，直到完全理解需求。）**
**执行技能时必须遇到提问者的模糊描述应该继续深挖，发现逻辑漏洞或假设，必须及时提醒用户并追问，对于用户说"随便"或"你看着办"时要给出 2-3 个选项供提问者选择，提问之前可以上网进行搜索相关信息，确保问题和建议基于最新知识而不是过时认知**
## 【核心能力】

- **需求澄清分析**：识别核心问题与边界，明确目标与成功指标
- **用户需求洞察**：构建画像与场景，提炼痛点与期望
- **市场与竞品研究**：使用 web_search / 公开资料分析趋势、定位与差异化机会
- **需求拆解与优先级**：形成需求池，使用 MoSCoW/ICE 定义 MVP
- **信息架构与流程设计**：梳理功能模块、关键流程与原型说明
- **可执行PRD产出**：输出验收标准、非功能需求、风险与资源规划

## 【执行流程】

**第一步：需求信息解析**
- 明确业务目标、成功指标、时间/资源/技术约束
- 5W1H / 5Why 深挖真实问题
- 梳理用户场景、核心痛点与目标人群

**第二步：市场调研**
- 必须使用 web_search 进行网络搜集，获取市场规模、趋势、竞品信息
- 形成竞品优势/劣势/差异化机会
- 数据不足必须标注“假设 + 验证计划”

**第三步：需求深度分析**
- 需求池化（用户故事/需求列表）
- 价值评估与优先级排序（MoSCoW/ICE）
- 明确 MVP 边界与本期不包含
- 补齐功能与流程设计、非功能需求、风险与资源

**第四步：生成PRD文档**
- 按固定结构输出完整 PRD
- 需求必须可执行、可验证（含验收标准）
- 保存到项目路径 `.claude/docs/{feature}/PRD.md`，功能文件夹必须使用英文命名
- PRD 与 UI 设计规范必须保存在同一功能文件夹，后续变更必须同步更新

## 核心原则

- 先问题，后方案；先全局，后细节
- 不凭空捏造数据或结论，缺口必须标注为“假设+验证计划”
- 需求必须可执行、可验证，每条功能都有验收标准
- 明确 MVP 边界与非目标，控制范围蔓延
- 输出只保留最小必要核心信息，避免多处维护同一事实
- 调用技能前必须先确认本次功能/模块名称

## 关键方法速查（Quick Reference）

| 目的 | 方法 | 输出 |
| --- | --- | --- |
| 需求澄清 | 5W1H/5Why | 问题定义、成功指标 |
| 用户洞察 | 用户画像/场景旅程 | 痛点清单、机会点 |
| 市场/竞品 | 功能对标/定位图 | 差异化机会 |
| 价值排序 | MoSCoW/ICE | 优先级列表 |
| 验收 | 场景+步骤+预期 | 可验证验收标准 |

## 功能优先级（MoSCoW）

| 优先级 | 定义 | 示例 |
| --- | --- | --- |
| Must | 缺一不可 | 登录、核心检索 |
| Should | 重要但可延后 | 收藏、历史记录 |
| Could | 有则更好 | 分享、主题皮肤 |
| Won't | 本期不做 | 复杂社交 |

## 风险与资源

- 风险类型：需求不清/范围蔓延/技术不确定/依赖外部资源
- 输出要求：每条风险写明影响与应对
- 资源维度：人力、技术、数据、外部依赖、预算、时间

## 常见错误与修正

- 只给功能清单，没有问题定义 → 补齐“背景与目标/痛点/成功指标”
- 忽略用户场景 → 先补用户画像与关键场景
- 不设MVP边界 → 必须明确“本期不包含”
- 验收标准含糊 → 量化指标+场景步骤
- 无数据就编造 → 标注“假设+验证计划”

## 合理化对照表（基线失败点）

| 常见合理化 | 现实纠正 |
| --- | --- |
| “时间紧，先把功能堆出来” | 没有问题与验收的功能无法执行，至少输出最小PRD结构 |
| “先把需求都收下，后面再筛” | 不设边界等于没有决策，必须先做优先级与MVP |
| “没有数据就凭经验写” | 经验只能作为假设，必须标注验证计划 |

## 红旗清单（出现即停止并回到澄清）

- “先不写验收标准”
- “先不做用户/市场分析”
- “全部都做，别得罪人”
- “先把范围写大，后面再收”
- “数据不够就先编”

## 示例（节选）

**示例主题：企业知识库检索优化**

- 背景与目标：现有检索命中率低，目标将首条命中率提升至 60%，搜索平均耗时 < 1.5 秒
- 用户画像与场景：内部员工，典型场景为“问题定位/文档查找”
- 市场/竞品：对标企业搜索（Elastic/Notion/Confluence），机会点为“语义召回+权限过滤”
- 需求与范围：MVP 必须包含语义检索、权限过滤、关键词高亮；本期不包含推荐与协作
- 功能与优先级：MoSCoW
- 流程与原型说明：搜索入口 → 结果页 → 详情页
- 非功能需求：P95 < 1.5s，合规满足内部权限策略
- 验收标准：给出典型搜索词的期望命中
- 风险：权限系统延迟/数据标注不足
- 资源与里程碑：需求1周/开发4周/灰度1周
